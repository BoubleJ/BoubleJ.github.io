# 개요

웹접근성을 준수하는 일은 특정 사용자 집단을 위한 선택 사항이 아니라, 모든 사용자를 위한 기본적인 품질 기준입니다. 겉으로는 단순한 탭 UI처럼 보이더라도, 스크린리더와 키보드 사용자에게 동등한 경험을 제공하려면 몇 가지 핵심 웹접근성 패턴을 의도적으로 적용해야 합니다.

이 글에서는 실제 프로젝트를 진행하며 웹접근성을 조금 더 꼼꼼하게 반영했던 사례를, 실제 코드와 UI 스크린샷을 바탕으로 어떤 접근성 원칙을 어떻게 코드로 구현했는지 정리해보려 합니다.



# 탭 메뉴 컴포넌트

![관심장르주간랭킹스와이퍼탭](/image/관심장르주간랭킹스와이퍼탭.png)

위 이미지는 사내 서비스 메인 화면에 노출되는 탭 메뉴 섹션입니다. 이 UI에서 어떤 웹접근성 원칙을 어떻게 코드로 지켰는지, 실제 구현 코드와 함께 살펴보겠습니다.


## 탭 UI의 WAI-ARIA 패턴: role과 aria-* 속성

탭 영역을 보면, **WAI-ARIA 탭 패턴**을 따르고 있습니다.

```tsx
return (
  <HeadingBox
    id={id}
    title={title}
    link={link ?? ''}
    className={className}
  >
    <div className={cn(LIST_TAB_CLASS)} role="tablist">
      {tabList.map((item, index) => {
        const selected = item === selectedTab
        return (
            <Button
              variant={selected ? 'solid-round' : 'outline-round'}
              size="xsmall"
              color={selected ? 'sub-green-01' : 'bright-01'}
              className={cn(BTN_TAB_CLASS)}
              onClick={() => setSelectedTab(item)}
              id={`${id}Tab${index}`}
              role="tab"
              aria-selected={selected}
              aria-controls={panelId}
            >
              {item}
            </Button>
        )
      })}
    </div>
    <div id={panelId} role="tabpanel" aria-labelledby={selectedTabId} className="min-height-192">
      {panel}
    </div>
  </HeadingBox>
)
```

#### `role="tablist"`로 탭 컨테이너의 역할 정의

```tsx
<div className={cn(LIST_TAB_CLASS)} role="tablist">
```

시각적으로는 단순한 버튼 묶음이지만, 스크린리더에게는 **“이 안에 탭들이 있다”**는 의미를 전달해야 합니다.  
`role="tablist"`를 지정함으로써, 보조기기 입장에서는 이 영역이 “탭 컨트롤들의 리스트”라는 것을 인식하게 됩니다.

#### 각 버튼에 `role="tab"` 부여

```tsx
<Button
  ...
  id={`${id}Tab${index}`}
  role="tab"
  aria-selected={selected}
  aria-controls={panelId}
>
  {item}
</Button>
```

- 시멘틱하게는 `button`이지만, 이 UI에서는 버튼들이 서로 **상호 배타적인 “탭” 역할**을 합니다.
- 그래서 `role="tab"`을 추가해, 이 컨트롤이 단순 버튼이 아니라 “탭 컴포넌트의 한 항목”이라는 정보를 제공합니다.
- 스크린리더는 “탭 1/3, 기다리면 무료, 선택됨”처럼 읽어주게 됩니다(툴/설정에 따라 표현은 조금씩 다름).

#### `role="tabpanel"`과 `aria-labelledby`로 패널과 탭 연결

```tsx
const panelId = `${id}Panel`
const selectedTabId = selectedTabIndex >= 0 ? `${id}Tab${selectedTabIndex}` : undefined

...

<div id={panelId} role="tabpanel" aria-labelledby={selectedTabId} className="min-height-192">
  {panel}
</div>
```

- 콘텐츠 영역은 `role="tabpanel"`로 선언해, **탭에 의해 제어되는 패널**임을 명시합니다.
- `aria-labelledby={selectedTabId}`로 현재 선택된 탭의 `id`를 참조함으로써, 스크린리더는 이 패널을 “(선택된 탭의 레이블)에 대응하는 패널”로 이해합니다.

#### `aria-controls`로 탭 → 패널 방향 링크

```tsx
aria-controls={panelId}
```

- 반대로 탭에서는 `aria-controls`를 통해 **자신이 제어하는 패널의 id**를 알려줍니다.
- 이로써 탭과 패널은 양방향으로 연결됩니다.
  - 탭: “나는 이 패널을 제어해”
  - 패널: “나는 이 탭에 의해 라벨링되어 있어”

이 패턴 덕분에, 시각 정보 없이도 “어떤 탭을 선택하면, 어떤 콘텐츠가 바뀌는지”를 보조기기가 안정적으로 추론할 수 있습니다.



### 상태 관리와 시각적·비시각적 피드백의 일치

탭 UI에서 중요한 것은 “어떤 탭이 선택되어 있는지”를 모든 사용자에게 **일관되게** 알려주는 것입니다.

```tsx
const [selectedTab, setSelectedTab] = useState<FreeCouponTabTitle>('기다리면 무료')

...

const selected = item === selectedTab

<Button
  variant={selected ? 'solid-round' : 'outline-round'}
  size="xsmall"
  color={selected ? 'sub-green-01' : 'bright-01'}
  ...
  aria-selected={selected}
>
  {item}
</Button>
```

여기서는 하나의 상태 `selectedTab`이 두 가지 층위에 동시에 반영됩니다.

- **시각적 층위**  
  - `variant`: 선택된 탭은 `solid-round`, 나머지는 `outline-round`.  
  - `color`: 선택된 탭은 `sub-green-01`, 비선택 탭은 `bright-01`.  
  - 사용자는 한눈에 “지금 어떤 탭이 활성화인지”를 색과 배경으로 알 수 있습니다.
- **보조기기 층위**  
  - `aria-selected={selected}`로, 같은 정보를 스크린리더에 전달합니다.  
  - 시각적으로 강조된 탭과 스크린리더가 “선택됨”이라고 읽는 탭이 완전히 일치합니다.

이렇게 **단일 소스 상태(`selectedTab`)**를 기준으로 시각적 표현과 WAI-ARIA 속성이 함께 갱신되기 때문에,  
시각 사용자와 비시각 사용자 모두에게 **동일한 상태 정보**가 전달됩니다.

### WCAG 기준과 연결해 보기

이번 구현을 WCAG(Web Content Accessibility Guidelines)의 관점에서 다시 정리해 보면 다음과 같습니다.

- **지각 가능(Perceivable)**  
  - `HeadingBox`의 `title`로 섹션의 목적을 명확히 전달합니다.  
  - 탭의 텍스트 레이블(예: “기다리면 무료”)은 탭의 기능을 직관적으로 설명합니다.
- **운용 가능(Operable)**  
  - 각 탭은 실제로 `Button` 컴포넌트를 사용해 구현되어, 기본적으로 키보드 포커스를 받을 수 있고 `Enter`/`Space`로 활성화할 수 있습니다.  
  - `role="tab"`, `role="tabpanel"`과 `aria-*` 관계 설정으로 탭-패널 구조를 보조기기가 정확히 이해할 수 있습니다.
- **이해 가능(Understandable)**  
  - 탭과 패널의 관계가 WAI-ARIA 패턴대로 연결되어 있어, 스크린리더가 “지금 어디에 있는지, 어떤 콘텐츠가 바뀌는지”를 일관되게 설명할 수 있습니다.
- **견고성(Robust)**  
  - 표준화된 ARIA 패턴을 사용했기 때문에, 다양한 스크린리더·브라우저 조합에서도 일관된 결과를 기대할 수 있습니다.



### 마무리: 작은 UI에서도 큰 접근성 차이를

이번 예시는 메인 페이지의 작은 **탭 UI 한 영역**만을 다루지만,  
여기서 적용한 패턴은 다른 곳에서도 그대로 재사용할 수 있는 **접근성 설계 템플릿**입니다.

- 시맨틱 역할(`role`)과 관계(`aria-controls`, `aria-labelledby`)를 명시하고,  
- 단일 상태(`selectedTab`)를 기준으로 시각 표현과 ARIA 상태를 함께 갱신하며,  
- 섹션 제목(`HeadingBox`)으로 영역의 목적을 분명히 해주는 것.

이 세 가지만 지켜도, 스크린리더 사용자와 키보드 사용자에게 훨씬 친절한 UI를 제공할 수 있습니다.  
앞으로는 여기에 한 걸음 더 나아가, **좌우 방향키로 탭 이동**, **Home/End 키로 첫/마지막 탭 이동** 같은 키보드 인터랙션까지 추가해  
WAI-ARIA 탭 패턴을 완성형에 가깝게 구현해보는 것도 좋은 다음 단계가 될 것입니다.

```ts
import * as React from 'react'
import { Drawer as DrawerPrimitive } from 'vaul'
import { cn } from '@/src/common/utils/twMerge'

const Drawer = ({ shouldScaleBackground = true, ...props }: React.ComponentProps<typeof DrawerPrimitive.Root>) => (
  <DrawerPrimitive.Root shouldScaleBackground={shouldScaleBackground} {...props} />
)
```



```ts
export default function SidebarProvider({ children }: { children: React.ReactNode }) {
  return (
    <Drawer direction="right" autoFocus>
      {children}
    </Drawer>
  )
}

```
